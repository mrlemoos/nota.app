## Package manager and dependency catalogue

- **pnpm only** at the repo root ([`package.json`](package.json) **`packageManager`**). Use **`pnpm install`**, **`pnpm exec nx …`**, **`pnpm add -wD …`**. Do not reintroduce **`package-lock.json`** or **`npm ci`** for this workspace. CI, Vercel, and Docker use **`pnpm install --frozen-lockfile`** (with **`corepack enable pnpm`** where needed). The workspace **`storeDir`** is **repo-local** ([`pnpm-workspace.yaml`](pnpm-workspace.yaml) → **`.pnpm-store/`**, gitignored); disk use grows with dependencies like any pnpm store.
- **Catalogue:** Shared versions for external deps used in **more than one** of **`packages/*`** or **`apps/*`** live under **`catalog`** in [`pnpm-workspace.yaml`](pnpm-workspace.yaml); every such consumer must reference **`"catalog:"`** in its **`package.json`** (not divergent semver ranges). Internal **`@nota.app/*`** packages use **`workspace:*`**.
- **Hoisting:** [`.npmrc`](.npmrc) **`node-linker=hoisted`** keeps **`node_modules`** layout compatible with **electron-builder** (see [`apps/nota-electron/electron-builder.yml`](apps/nota-electron/electron-builder.yml)).

- **Dev:** **`pnpm run dev`** = `nx run-many -t dev`—parallel (notes **http://localhost:4200**, marketing, `apps/nota-server`, Electron). Single app: **`pnpm exec nx dev <project>`** (e.g. **`pnpm exec nx dev @nota.app/nota.app`**). Electron-only: **`pnpm run electron:dev`** (`nx dev nota-electron`).

## Git commits (enforced)

Must use [Conventional Commits](https://www.conventionalcommits.org/).

- Subject: `<type>(optional scope): description`—types `feat|fix|docs|style|refactor|perf|test|build|ci|chore|…`; imperative; ~72 chars; lowercase after colon unless proper noun.
- Body when useful; **`BREAKING CHANGE:`** footer if incompatible change.
- Reject vague subjects (`update`, `fixes`, `wip`, …); unrelated changes → split commits or scoped `chore`. Committing for user: **stage by path** if mixed.

## General Guidelines for working with Nx

- Explore workspace → **`nx-workspace`** skill first (projects, targets, deps).
- Run tasks via **`nx`** (`nx run`, `nx run-many`, `nx affected`), not raw tooling—prefix with package manager (`pnpm exec nx …`).
- Use Nx MCP when helpful. Plugin docs: `node_modules/@nx/<plugin>/PLUGIN.md`. **Never** guess CLI flags—**`nx_docs`** or **`--help`**.

## Scaffolding & Generators

- Scaffold apps/libs → **`nx-generate`** skill **before** MCP/explore.

## When to use nx_docs

- **Use:** advanced nx config, unfamiliar flags, migrations, plugins, edge cases.
- **Skip:** basic `nx g`, routine commands—**nx-generate** handles generator discovery.
