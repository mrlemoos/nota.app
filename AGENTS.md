# Agent memory (nota.app)

## Workspace and stack

- Primary app is `apps/nota.app`: **Vite** client SPA (`index.html` → `app/main.tsx` → `SpaApp`), React 19, Nx monorepo; UI primitives from `@/components/ui/*` (Base UI–backed shadcn-style) and semantic theme tokens (`bg-background`, `muted`, `foreground`, etc.)—avoid hard-coded light neutrals on the notes surface. **`apps/nota.app` is client-only:** no Node route handlers, no `CLERK_SECRET_KEY`, no server-side OG fetch—those live in **`apps/nota-server`**.
- Notes use Supabase Postgres with RLS; **identity is Clerk** (JWT to Supabase via third-party auth); the browser uses `@supabase/supabase-js` with `accessToken` from Clerk session tokens. Data access through typed helpers in `app/models/notes.ts` rather than ad hoc queries in UI.
- TipTap edits run client-only; note body is stored as ProseMirror JSON in a `jsonb` column. The editor sync effect compares the `content` prop to the live document with ProseMirror `Node.fromJSON` + `doc.eq` (not `JSON.stringify`), since `jsonb` can reorder keys and a false diff would call `setContent` and move the caret. Also avoid pushing stale `content` into parent state after `updateNote`: the `.select()` row can lag local edits (typing during an in-flight body save, or a title save before the body debounce flushes), so merge the latest local body snapshot into the note passed to `onNoteUpdated` instead of trusting the returned row’s `content` alone.
- Public env for the app: `VITE_CLERK_PUBLISHABLE_KEY`, `VITE_SUPABASE_URL`, and `VITE_SUPABASE_ANON_KEY` (anon key only; see `apps/nota.app/.env.example`). Server-only **`CLERK_SECRET_KEY`** is used only on **`apps/nota-server`** (Bearer `verifyToken`, cookie/session handling where applicable, **Clerk Billing** subscription checks, **`GET /api/og-preview`**, and assistive **`POST /api/audio-to-note`**). The Vercel **`apps/nota.app`** project is a **static SPA** only—**do not** rely on Vercel serverless `api/*` for OG or auth; there is no deployed serverless layer in that package. **`VITE_NOTA_SERVER_API_URL`** (required for Nota Pro, link previews, `GET /api/og-preview`, and assistive audio-to-note from the browser): calls go to **`apps/nota-server`** with `Authorization: Bearer <Clerk session JWT>`; if the URL is missing at runtime, the client treats the user as not entitled without calling the network, and **`fetchOgPreviewForEditor`** throws until the URL is set. Never expose `CLERK_SECRET_KEY` or Supabase service keys to the client. SQL migrations live under `supabase/migrations/` at the repo root (including Clerk third-party RLS: `0008_clerk_third_party_auth.sql`).
- Root `.gitignore` should include `supabase/.temp/`—Supabase CLI local metadata there can contain connection strings and must not be committed.
- **Stack discipline:** default stack is Nx + Vite + React (`apps/nota.app`) and Astro (`apps/nota-marketing`). Do not introduce a new framework or drive a major dependency upgrade unless the change is justified by a **user-visible** improvement or **security** fix (spell that out in the PR).
- **Local dev (root):** **`npm run dev`** runs **`nx run-many -t dev`**, which starts the **`dev`** target on every Nx project that defines it **in parallel** (typically the notes SPA Vite server on **`http://localhost:4200`**, marketing Astro, **`apps/nota-server`**, and Electron dev as configured). For one app only, use **`npx nx dev <project>`** (for example **`npx nx dev @nota.app/nota.app`**). To run only the Electron dev target (without every other **`dev`** process), use **`npm run electron:dev`** (`nx dev nota-electron`).

## Git commits (enforced)

Agents **must** record changes with [Conventional Commits](https://www.conventionalcommits.org/). Do not commit with a bare description, vague wording, or a subject that omits the type prefix.

- **Subject (required):** `<type>(optional scope): description`
  - **Type:** one of `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore` (or another well-known type from the spec if it clearly fits).
  - **Scope:** optional; use a short, real scope from the repo when it helps (e.g. `nota.app`, `nota-server`, `marketing`, `electron`).
  - **Description:** imperative mood, lowercase after the colon unless a proper noun, **~72 characters** for the subject line.
- **Body (when useful):** rationale, subtasks, migration notes, or “what changed / why” for non-obvious diffs.
- **Breaking changes:** include `BREAKING CHANGE:` in the footer when behaviour or APIs change incompatibly.

**Reject / avoid:** subjects like `update`, `fixes`, `wip`, `misc`, `changes`, or only a ticket id with no type; mixing unrelated concerns in one commit without a clear primary type (prefer **split commits** or a `chore` scope per area).

**When committing for the user:** use `git commit` with a message that already satisfies the above; if the working tree mixes unrelated changes, **stage by path** and **multiple commits** rather than one ambiguous subject.

## Notes UI and client state

- The signed-in **notes** experience lives in `NotesSpaShell` (`app/components/notes-spa-shell.tsx`); primary navigation uses **`app/lib/app-navigation.ts`** (location **hash**), not React Router nested routes.
- Sidebar open/closed is Zustand with `persist` to localStorage under the key `nota-notes-sidebar`.
- The `/notes` layout row fills the viewport (`h-dvh` with `min-h-0` on the shell, `aside`, and `main`); only `main` scrolls the document, and the sidebar note list scrolls inside its own `flex-1 min-h-0 overflow-y-auto` region so the window does not grow with note height. In Electron (`titleBarStyle: 'hiddenInset'` in `apps/nota-electron`), `useIsElectron()` (detects preload `window.nota` or UA; modern Electron often omits the Electron token from the UA) drives traffic-light clearance on `/notes`: sidebar header uses `pl-20` and safe-area top padding; when the sidebar is collapsed the expand control is `position: fixed` in a `min-h-[52px]` `items-center` strip (`pl-20`, safe-area `pt`) with `pointer-events-none` on the wrapper and `pointer-events-auto` on the button. `main` uses the same top padding whether the sidebar is open or closed so toggling does not shift scroll layout. Window dragging uses **`-webkit-app-region`**: global classes **`.electron-window-drag`** / **`.electron-window-no-drag`** in `styles.css`, a fixed **`ElectronWindowDragBand`** in `spa-app.tsx` (`app/components/electron-window-drag-band.tsx`) from the traffic-light inset (`left-20`), and **`no-drag`** plus raised **`z-index`** on overlapping notes header / sidebar toggle so controls stay clickable.
- There is no global top nav. Note Graph and Settings (`/notes/settings`) links live in the notes sidebar footer (`routes/notes.tsx`); theme, today’s-note shortcut, account email, and sign out are on the Settings page. When the inline document title scrolls out of the notes layout `main` (`overflow-auto`, not the window), a read-only centered title appears as a fixed overlay at the top of the viewport (`routes/notes.tsx`) driven by `app/context/sticky-doc-title.tsx` and an `IntersectionObserver` rooted on that `main`.
- `useStickyDocTitle` must not throw when `StickyDocTitleProvider` is missing (e.g. Vitest `createRoutesStub` without root `Layout`); use a no-op fallback so partial trees still render.
- Note list sidebar heading and the `/notes` empty-state title use serif (`font-serif` / Instrument Serif) with `tracking-normal`; the open note’s document title field stays sans.
- Note editor is a centered document column (no bordered card around TipTap); the inline title is a debounced, wrapping textarea with auto height (saved to DB `title`); empty display label matches `persistedDisplayTitle` in `app/lib/note-title.ts` (“Untitled Note”). Body autosave must update `content` only and must not overwrite `title` from the first paragraph of the editor.
- After successful client-side saves from `NoteEditor`, refresh list metadata via the notes data layer (e.g. `refreshNotesList` / context updates) so sidebar titles and dates stay in sync.
- Delete note: from the trash control on each sidebar row (with confirm) and from the Command Palette when a note is open—**not** from a control beside the document title. Implementation uses **`spaDeleteNoteById`** (`app/lib/spa-delete-note.ts`) so IndexedDB and Supabase stay aligned.
- Create note: **`spaCreateNote`** (`app/lib/spa-create-note.ts`) from the notes shell and Command Palette; keep navigation/hash updates consistent with **`app/lib/app-navigation.ts`** when inserting the new id.
- Command palette (`app/components/command-palette.tsx`) mounts from `app/signed-in-command-palette.tsx` under the signed-in notes tree. Cmd/Ctrl+K toggles it from anywhere (including the TipTap body and title field); it lists notes from **`NotesData`**, opens notes via hash navigation, creates/deletes through the SPA helpers above, and signs out via Clerk (`useClerk().signOut()` + hash navigation)—not React Router `Form` posts.
- Micro-motion uses **`gsap`** and **`@gsap/react`** via `app/lib/nota-motion.ts` (`usePrefersReducedMotion`, shared eases/durations). Command palette enter/exit is GSAP-driven with Base UI Dialog `actionsRef`, `preventUnmountOnClose()` on close, and `unmount()` after the exit timeline; notes sidebar open/close tweens the `<aside>` `maxWidth`/`opacity`; shared `Button` adds a short press scale. **`vite.config.mts`** sets `ssr.noExternal: ['gsap', '@gsap/react']` so Vite bundles them for SSR—otherwise Node would load GSAP’s ESM package from `node_modules` on Vercel and crash with “Cannot use import statement outside a module”. Avoid GSAP on the TipTap document surface (caret and performance).

## Learned User Preferences

- PDF attachments should live inline in the TipTap document (insert at the cursor as a `notePdf` block) rather than only in a separate footer panel.
- The open note’s inline document title textarea should use a heavy sans weight and `text-pretty` so large, multi-line titles read clearly.
- Link to another note from the editor body by typing `@` (filterable mention menu), not a **Link to note** toolbar control.
- Prefer a minimal attachment entry in the editor chrome: no **Insert PDF or image** button or long instructional hint in the toolbar row; rely on drag-and-drop for PDFs and images while still surfacing upload errors when needed.
- After inserting an internal note link from the `@` flow, continue in normal paragraph typing without extending the link mark (e.g. `setParagraph` and clear stored marks after insert).
- Product and public docs favour a calm, direct voice: position nota.app against compulsive attention-economy patterns in thinking tools (not a blanket anti-AI stance); **no generic AI-generated prose in the editor** and no engagement-bait UI. **Carve-out:** optional, clearly labelled **assistive capture** flows: **Generate study notes from recording** (creates a new note with a formatted title; **`noteAudio`** then generated blocks) or **Add study notes from recording to this note** when a note is open (keeps the current title; appends **existing body → `noteAudio` when present → generated blocks**; offline pending jobs carry **`append`** and skip the queued-title stub). Both transform **user-provided audio** into structured notes via **`apps/nota-server`** and xAI—distinct from ghostwriting, inline completion, or undifferentiated “AI writing” in the note body.
- Public-facing copy (marketing site, store listings) should match product behaviour: there is **no unpaid tier or free trial**—users download the Mac app, sign in, and **subscribe in-app** for full access, cloud sync, and backup; marketing must not promise a free full vault. Core purchase flows stay in-app after install, not as a web-only checkout on the notes surface. On **Settings**, the Nota Pro / subscription block should foreground **direct plan selection and checkout** (Clerk Billing UI) rather than long marketing copy.
- Notes should be offline-first / local-first: capture edits locally and sync to the cloud; when the app starts with network available, load notes and attachments from the server.
- Optional “today’s note” shortcut (Mod+D when enabled): no streaks, reminders, or other gamification; command palette can surface the same action with a matching hotkey hint when the setting is on. New daily notes use the local long date as the initial title (e.g. 4 March 2026), not “Untitled Note”.
- Shared `Button` is Base UI–backed and does not support Radix-style `asChild`; for controls that must behave as real links (e.g. billing portal URLs in new tabs), style an `<a>` with exported `buttonVariants` and `cn()`, or use `Button` + `window.open` if trading off middle-click and copy-link behaviour is acceptable.
- Keep the in-app and favicon mark minimal and in sync (`NotaLogo` ↔ `public/favicon.svg`); after changing geometry run **`npm run generate:nota-icons`** for `apple-touch-icon.png` and, on macOS, `apps/nota-electron/buildResources/icon.icns`.
- When tuning the **packaged** Mac app, treat **first-open note** latency as important (hosted SPA load plus editor hydration), not only the local Vite dev shell.

## Learned Workspace Facts

- **`NotesDataProvider`** (`app/context/notes-data-context.tsx`) calls **`fetchNotaProEntitled`** ([`app/lib/nota-server-client.ts`](apps/nota.app/app/lib/nota-server-client.ts): **`apps/nota-server`** at **`VITE_NOTA_SERVER_API_URL`** with Bearer JWT; missing base URL → 401-style handling without a network call) and exposes **`notaProEntitled`** (meaning **licensed / subscribed**) plus `refreshNotesList` / patch helpers. When **`notaProEntitled`** is true, it loads Supabase notes and `user_preferences`, merges with IndexedDB via `app/lib/notes-offline/`. When false, the user gets **no vault**: notes list stays empty (no IndexedDB bootstrap for unpaid users). Offline continuation uses **`readNotaServerEntitledSession`** so previously entitled users can still read local cache when the network or entitlement request fails. Signed-in but unpaid users see **`NotesSpaShell`** paywall UI (sidebar hidden, hash steered to **Settings** to subscribe). **`NoteDetailPanel`** / uploads / cloud saves require entitlement. Per-note **`editor_settings`** (`jsonb`) syncs when entitled and online. When entitled after load with an empty vault (`notes.length === 0`) and `user_preferences.welcome_seeded` is false, **`runWelcomeNoteSeedIfNeeded`** (`welcome-note-seed.ts`) seeds **`welcome-note-doc`** via **`createLocalOnlyNote`**, persists **`welcome_seeded`** with **`upsertUserPreferences`**, a per-user promise map to avoid Strict Mode double-insert, then **`refreshNotesList`** and **`setAppHash`** to open the welcome note. **Clerk Billing** server truth lives in **[`apps/nota-server/src/lib/clerk-billing.server.ts`](apps/nota-server/src/lib/clerk-billing.server.ts)** (cached; fail closed); checkout in **Settings** via **`PricingTable`** where enabled; after purchase call **`postNotaProInvalidate`** and **`refreshNotesList`**; configure plans in the **Clerk Dashboard** (Stripe under the hood; not synced from legacy RevenueCat). With **`CLERK_SECRET_KEY`** in **`apps/nota-server/.env`**, run **`npm run validate:billing`** (see **`apps/nota-server/README.md`**) to fetch Clerk Billing plans and compare public user prices / no-trial expectations to **`apps/nota-marketing`** guide USD constants and **`AGENTS.md`** policy. **`refreshNotesList({ silent: true })`** refreshes list data **without** flipping global **`loading`** / **`shellReady`**—use it for background work (offline outbox follow-ups, editor revalidate, create/delete tails, command palette) so **`useNotesOfflineSync`** (gated by **`shellReady`**) does not thrash; keep non-silent refresh for explicit user feedback (e.g. Settings “refresh subscription”); reset refresh sequencing when **`userId`** changes.
- Per-note attachments use Supabase Storage bucket `note-pdfs` (PDFs plus allowlisted raster images per migration), `note_attachments` rows, and TipTap `notePdf` / `noteImage` atom blocks in `notes.content` JSON (`attachmentId`, `filename`); generalised `noteAttachmentStoragePath` and shared upload helpers in `app/lib/pdf-attachment-client.ts`. Drag-and-drop uploads require **`notaProEntitled`** (`tiptap-editor.tsx`). RLS allows owners to update `note_attachments.filename` for display-name renames on PDF nodes (double-click the label). Image blocks resolve signed URLs and refresh before expiry; signed-URL logic should depend on stable fields such as `storage_path`, not the whole attachment row object, so loader revalidation (new object references) does not refetch and flicker the image. PDF modal preview uses PDF.js in a lazy-loaded chunk (`pdf-js-modal-preview.tsx`); keep PDF.js off the eager import path for `note-pdf-extension.tsx` (use `React.lazy` and `Suspense`) or `notePdf` nodes can vanish. Render the preview `<dialog>` with `createPortal` to `document.body`. `app/lib/pdf-preview-url.ts` adds `#toolbar=0&navpanes=0` for iframe fallback only when PDF.js fails. File insert uses drag-and-drop with ProseMirror `editorProps` drop handling on the real editor surface when no file picker row is shown. Deleting a note: `deleteNote` in `app/models/notes.ts` lists `note_attachments`, removes objects from Storage (`note-pdfs`) while the note row still exists (storage RLS), then deletes the note; attachment rows drop via `ON DELETE CASCADE`.
- Link previews: the SPA calls **`apps/nota-server`** **`GET /api/og-preview?url=`** with Bearer auth ([`app/lib/og-preview-client.ts`](apps/nota.app/app/lib/og-preview-client.ts); server-side fetch and parsing in [`apps/nota-server/src/lib/og-preview.server.ts`](apps/nota-server/src/lib/og-preview.server.ts)). The editor exposes a TipTap `linkPreview` block and debounced conversion of link-only paragraphs via `app/components/tiptap/link-preview-scan.ts`. If OG fetch fails or returns no usable title/description/image, revert to a normal paragraph with a link; the link mark uses `skipLinkPreview` so the scanner does not immediately re-promote the same URL and loop. Paragraphs whose only link href is an internal note path `/notes/<uuid>` are not promoted to `linkPreview`. **`VITE_NOTA_SERVER_API_URL`** must be set for link previews to work (there is no same-origin `/api/og-preview` on the static SPA). **`fetchOgPreview`** follows redirects with **`redirect: 'manual'`**, re-runs **`assertUrlSafeForOgFetch`** on each hop (bounded count), and checks resolved addresses via **`dns.lookup`** so private and link-local targets stay blocked; **`og:image`** is restricted to **`http:`/`https:`** before the JSON response, and the client applies **`safeOgImageSrcForPreview`** ([`app/lib/og-image-url.ts`](apps/nota.app/app/lib/og-image-url.ts)) before rendering **`<img src>`** so persisted attrs cannot revive disallowed schemes. In production, **`nota-server`** logs a **`console.warn`** when **`NOTA_SERVER_CORS_ORIGINS=*`** reflects arbitrary browser origins—prefer an explicit allowlist.
- Internal note links: `href` is path-only `/notes/<uuid>` (see `app/lib/internal-note-link.ts`); `NotaLink` uses `openOnClick: false` with a custom click handler—internal paths update the in-app hash / navigation, modifier-click opens a new tab, external URLs use `window.open`; `@` mention menus use the live notes list from editor/`NotesData` context. Forward/back indexes for the vault are derived from TipTap JSON via `app/lib/note-link-graph.ts` (no separate links table for v1); the graph surface is lazy-loaded React Flow, and `NoteBacklinksPanel` lists inbound links on the open note. The graph view (`notes-graph-view.tsx`) uses **`filterNotesForNoteGraph`** so notes with **`editor_settings.showInNoteGraph === false`** are omitted; **Show in note graph** in the note layout menu (`note-layout-menu.tsx`) toggles that flag (backlinks still list all notes).
- Mermaid in notes uses `NotaCodeBlock` (`app/components/tiptap/nota-code-block.tsx`): `StarterKit.configure({ codeBlock: false })` plus `CodeBlockLowlight` from `@tiptap/extension-code-block-lowlight` with a shared `notaLowlight` instance (`app/lib/nota-lowlight.ts`: `createLowlight(common)` and register `mermaid` as plaintext so diagram fences are not auto-highlighted); React node view for the block. When `attrs.language` is `mermaid` (case-insensitive), lazy-load `mermaid`, debounce renders, use `securityLevel: 'strict'`, and derive light/dark theme from the `<html>` `dark` class. Non-mermaid fences get lowlight `hljs-*` decorations; style tokens in `styles.css` under `.tiptap-editor .tiptap-code-block` for light and `.dark`. Fence input rules allow broader language ids (e.g. `c++`) than stock TipTap. In `TipTapEditor`, declare hooks such as `useCallback` for toolbar actions before any early return that skips the mounted editor tree so hook order stays valid.
- TipTap body sync and editor instance stability: apply `content` from props with `setContent` when `noteId` changes (user opened a different note), not on every parent update after save or list refresh for the same note—otherwise the selection can jump. **Exception:** server-driven replacement on the **same** `noteId` from assistive study capture (typically a leading **`noteAudio`** in the **new-note** path, or appended **`noteAudio`** after existing content in **append-to-open-note**) must still reach the editor—advance a **`contentRevision`** tied to the row (e.g. **`note.updated_at`**) and reapply when the document differs; keep **`NoteDetailPanel`** on the newer **`notes`** list row (merge with local) and **refetch `note_attachments`** so `noteAudio` resolves in **`attachmentsById`**. After a successful body save, align `lastSavedContent` with the merged local body snapshot passed to `onNoteUpdated`, not only the payload that triggered the request. `useEditor` depends on `extensions`; callbacks captured in the `extensions` `useMemo` (e.g. `onSaveDueDate`) must stay referentially stable when parent `note` updates after saves—otherwise `extensions` is recreated, the editor instance is destroyed, and the caret jumps. ## General Guidelines for working with Nx

- For navigating/exploring the workspace, invoke the `nx-workspace` skill first - it has patterns for querying projects, targets, and dependencies
- When running tasks (for example build, lint, test, e2e, etc.), always prefer running the task through `nx` (i.e. `nx run`, `nx run-many`, `nx affected`) instead of using the underlying tooling directly
- Prefix nx commands with the workspace's package manager (e.g., `pnpm nx build`, `npm exec nx test`) - avoids using globally installed CLI
- You have access to the Nx MCP server and its tools, use them to help the user
- For Nx plugin best practices, check `node_modules/@nx/<plugin>/PLUGIN.md`. Not all plugins have this file - proceed without it if unavailable.
- NEVER guess CLI flags - always check nx_docs or `--help` first when unsure

## Scaffolding & Generators

- For scaffolding tasks (creating apps, libs, project structure, setup), ALWAYS invoke the `nx-generate` skill FIRST before exploring or calling MCP tools

## When to use nx_docs

- USE for: advanced config options, unfamiliar flags, migration guides, plugin configuration, edge cases
- DON'T USE for: basic generator syntax (`nx g @nx/react:app`), standard commands, things you already know
- The `nx-generate` skill handles generator discovery internally - don't call nx_docs just to look up generator syntax
