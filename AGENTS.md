# Agent memory (nota.app)

## Workspace and stack

- **Workspace packages** under [`packages/`](packages/) (npm `workspaces`: `apps/*`, `packages/*`): shared modules — `@nota.app/clerk-oauth-protocol` (desktop OAuth URL constants; dual export Vite vs Node), `@nota.app/database-types`, `@nota.app/validation` (Zod), `@nota.app/nota-server-client` (Nota Pro fetch helpers; app wires env + Clerk), `@nota.app/notes-offline` (IndexedDB + merge/outbox; app [`src/lib/notes-offline.ts`](apps/nota.app/src/lib/notes-offline.ts) re-exports package + [`notes-offline-sync.ts`](apps/nota.app/src/lib/notes-offline-sync.ts) for Supabase drain). TipTap extensions and `@/components/ui` stay in **`apps/nota.app`** (tight coupling to shell, context, Tailwind/shadcn).
- Primary app `apps/nota.app`: **Vite** client SPA (`index.html` → `src/main.tsx` → `NotaApp`), React 19, Nx monorepo; UI primitives from `@/components/ui/*` (Base UI–backed shadcn-style) and semantic theme tokens (`bg-background`, `muted`, `foreground`, etc.)—avoid hard-coded light neutrals on notes surface. **`apps/nota.app` is client-only:** no Node route handlers, no `CLERK_SECRET_KEY`, no server-side OG fetch—those live in **`apps/nota-server`**.
- **Client source tree** is rooted at **`apps/nota.app/src/`**. Vite and `tsconfig` map **`~/`** and **`@/`** to that directory (not a top-level `app/` folder). Hash UI state, session, and notes shell are composed in `app-root.tsx` (`NotaApp`) and `components/notes-shell.tsx` (`NotesShell`); vault create/delete flows use `clientCreateNote` / `clientDeleteNoteById` in `lib/create-note-client.ts` and `lib/delete-note-client.ts`.
- Notes use Supabase Postgres with RLS; **identity is Clerk** (JWT to Supabase via third-party auth); browser uses `@supabase/supabase-js` with `accessToken` from Clerk session tokens. Data access via typed helpers in `src/models/notes.ts`, not ad hoc queries in UI.
- TipTap edits run client-only; note body stored as ProseMirror JSON in `jsonb` column. Editor sync effect compares `content` prop to live document with ProseMirror `Node.fromJSON` + `doc.eq` (not `JSON.stringify`)—`jsonb` can reorder keys, false diff calls `setContent` and moves caret. Avoid pushing stale `content` into parent state after `updateNote`: `.select()` row can lag local edits (typing during in-flight body save, or title save before body debounce flushes)—merge latest local body snapshot into note passed to `onNoteUpdated` instead of trusting returned row's `content`.
- Public env for app: `VITE_CLERK_PUBLISHABLE_KEY`, `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (anon key only; see `apps/nota.app/.env.example`). Server-only **`CLERK_SECRET_KEY`** used only in **`apps/nota-server`** (Bearer `verifyToken`, cookie/session handling, **Clerk Billing** subscription checks, **`GET /api/og-preview`**, assistive **`POST /api/audio-to-note`**). Vercel **`apps/nota.app`** project is **static SPA** only—**no** Vercel serverless `api/*` for OG or auth. **`VITE_NOTA_SERVER_API_URL`** (required for Nota Pro, link previews, `GET /api/og-preview`, assistive audio-to-note): calls go to **`apps/nota-server`** with `Authorization: Bearer <Clerk session JWT>`; missing URL → client treats user as not entitled without network call, **`fetchOgPreviewForEditor`** throws. Never expose `CLERK_SECRET_KEY` or Supabase service keys to client. SQL migrations in `supabase/migrations/` at repo root (incl. Clerk third-party RLS: `0008_clerk_third_party_auth.sql`).
- Root `.gitignore` must include `supabase/.temp/`—Supabase CLI local metadata can contain connection strings, must not be committed.
- **Stack discipline:** default stack is Nx + Vite + React (`apps/nota.app`) and Astro (`apps/nota-marketing`). No new framework or major dependency upgrade unless justified by **user-visible** improvement or **security** fix (spell out in PR).
- **Local dev (root):** **`npm run dev`** runs **`nx run-many -t dev`**—starts **`dev`** target on every Nx project in parallel (notes SPA Vite on **`http://localhost:4200`**, marketing Astro, **`apps/nota-server`**, Electron dev). Single app: **`npx nx dev <project>`** (e.g. **`npx nx dev @nota.app/nota.app`**). Electron-only: **`npm run electron:dev`** (`nx dev nota-electron`).

## Git commits (enforced)

Agents **must** record changes with [Conventional Commits](https://www.conventionalcommits.org/). No bare descriptions, vague wording, or subjects missing type prefix.

- **Subject (required):** `<type>(optional scope): description`
  - **Type:** `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore` (or other well-known spec type).
  - **Scope:** optional; short, real repo scope (e.g. `nota.app`, `nota-server`, `marketing`, `electron`).
  - **Description:** imperative mood, lowercase after colon unless proper noun, **~72 chars** subject line.
- **Body (when useful):** rationale, subtasks, migration notes, "what/why" for non-obvious diffs.
- **Breaking changes:** `BREAKING CHANGE:` in footer when behavior or APIs change incompatibly.

**Reject/avoid:** subjects like `update`, `fixes`, `wip`, `misc`, `changes`, ticket-id-only; mixing unrelated concerns without clear primary type (prefer **split commits** or `chore` scope per area).

**When committing for user:** satisfy above; if working tree mixes unrelated changes, **stage by path** + **multiple commits** over one ambiguous subject.

## Learned User Preferences

- PDF attachments: inline in TipTap (`notePdf` block at cursor), not only in separate footer panel.
- Open note's inline document title textarea: heavy sans weight + `text-pretty` for large multi-line titles.
- Link to another note from editor body by typing `@` (filterable mention menu), not toolbar control.
- Minimal attachment entry in editor chrome: no **Insert PDF or image** button or instructional hint in toolbar; rely on drag-and-drop; surface upload errors when needed.
- After inserting internal note link from `@` flow, continue in normal paragraph without extending link mark (e.g. `setParagraph` + clear stored marks after insert).
- Product/public docs: calm, direct voice; position nota.app against compulsive attention-economy patterns in thinking tools (not blanket anti-AI). No generic AI-generated prose in editor, no engagement-bait UI. **Carve-out:** optional, clearly labelled **assistive capture** flows: **Generate study notes from recording** (new note with formatted title; **`noteAudio`** + generated blocks) or **Add study notes from recording to this note** (keeps title; appends existing body → `noteAudio` if present → generated blocks; offline pending jobs carry **`append`**, skip queued-title stub). Both transform **user-provided audio** into structured notes via **`apps/nota-server`** + xAI—distinct from ghostwriting or inline completion.
- Public-facing copy: **no unpaid tier or free trial**—users download Mac app, sign in, **subscribe in-app** for full access, cloud sync, backup; marketing must not promise free vault. Core purchase flows stay in-app. Settings subscription block: foreground **direct plan selection + checkout** (Clerk Billing UI), not long marketing copy.
- Notes offline-first / local-first: capture edits locally, sync to cloud; on start with network, load notes + attachments from server.
- Optional "today's note" shortcut (Mod+D when enabled): no streaks, reminders, gamification; command palette surfaces same action with matching hotkey hint. New daily notes use local long date as initial title (e.g. 4 March 2026), not "Untitled Note".
- Shared `Button` is Base UI–backed, no Radix-style `asChild`; for real links (e.g. billing portal URLs in new tabs), style `<a>` with `buttonVariants` + `cn()`, or use `Button` + `window.open` if middle-click/copy-link tradeoff acceptable.
- Keep in-app + favicon mark minimal + in sync (`NotaLogo` ↔ `public/favicon.svg`); after geometry changes run **`npm run generate:nota-icons`** for `apple-touch-icon.png` and on macOS `apps/nota-electron/buildResources/icon.icns`.
- For packaged Mac app, treat **first-open note** latency as important (hosted SPA load + editor hydration), not only local Vite dev shell.

## General Guidelines for working with Nx

- For navigating/exploring workspace, invoke `nx-workspace` skill first—has patterns for querying projects, targets, dependencies
- When running tasks (build, lint, test, e2e, etc.), always prefer `nx` (`nx run`, `nx run-many`, `nx affected`) over underlying tooling directly
- Prefix nx commands with workspace's package manager (e.g., `pnpm nx build`, `npm exec nx test`)—avoids globally installed CLI
- You have access to Nx MCP server + tools, use them
- For Nx plugin best practices, check `node_modules/@nx/<plugin>/PLUGIN.md` (not all plugins have this—proceed without if unavailable)
- NEVER guess CLI flags—check nx_docs or `--help` first when unsure

## Scaffolding & Generators

- For scaffolding tasks (creating apps, libs, project structure), ALWAYS invoke `nx-generate` skill FIRST before exploring or calling MCP tools

## When to use nx_docs

- USE for: advanced config options, unfamiliar flags, migration guides, plugin configuration, edge cases
- DON'T USE for: basic generator syntax (`nx g @nx/react:app`), standard commands, things you already know
- `nx-generate` skill handles generator discovery internally—don't call nx_docs just to look up generator syntax
